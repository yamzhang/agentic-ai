# 飞书多维表格字段映射

本项目最终落飞书时，固定采用两表模型：
- `客户信息`
- `商机快照`

`feishu_bitable_payload` 就是 CRM 理解结果与飞书多维表格写入之间的桥梁。

---

## 1. 客户信息

用途：
- 沉淀长期客户画像
- 按客户身份做更新

建议字段：
- 客户ID
- 客户名称
- 客户公司
- 职务
- 行业
- MBTI
- 是否单身
- 沟通风格
- 成交阻力
- 价格敏感程度
- 风险顾虑
- 客户画像摘要
- 客户负责人
- 最后更新时间
- 数据来源

### 写入策略
- 逻辑主键：`客户ID`
- 写入模式：`upsert`
- 一位客户只保留一条主记录

### 当前项目的关键规则
#### 弱值不覆盖强值
如果本轮值属于以下弱值，不覆盖已有明确旧值：
- `未明确`
- `未知`
- `待确认`
- 空值

#### 合并字段
以下字段采用“旧值保留 + 新值补充 + 去重”策略：
- `沟通风格`
- `风险顾虑`

---

## 2. 商机快照

用途：
- 每次会议结束后追加一条快照
- 保留完整推进轨迹

建议字段：
- 商机ID
- 客户ID
- 客户名称
- 客户公司
- 机会名称
- 商机描述
- 当前阶段
- Lead Score
- 意向等级
- 高净值优先
- 销售区域
- 业务价值
- 推荐动作
- 最新进展
- 下次跟进时间
- 最近会议时间
- 商机负责人
- 数据来源

### 写入策略
- 写入模式：`append`
- 每次会议结束后新增一条
- 不覆盖旧记录

---

## 3. 数据格式建议

### 3.1 文本字段
- 多值字段统一拼成 `；` 分隔的中文字符串，便于单元格展示
- `客户画像摘要`、`推荐动作`、`最新进展` 保持短文本

### 3.2 时间字段
当前项目在真实写飞书 Bitable 时，时间字段应优先转换成**毫秒时间戳**，尤其是：
- `最后更新时间`
- `下次跟进时间`
- `最近会议时间`

如果只是中间 JSON 结果，也可能先保留 ISO 8601 字符串；但真正写飞书前，应按字段类型做转换。

### 3.3 布尔 / 数值字段
- `Lead Score` 保持数值型
- `高净值优先` 保持布尔型

### 3.4 业务价值
- 有明确金额或区间时，按证据写入
- 如果会话中没有明确信号，可写 `暂无明确业务价值`

---

## 4. 实践建议

- 不要把原始 transcript 全量写进主表
- transcript 原文、录音链接、会议纪要全文更适合放外部存储或单独链接字段
- 真正落表前，先输出标准 CRM JSON，再映射成飞书 API 需要的字段格式
- 如果有多客户会议，客户信息应支持多行更新；商机快照通常仍是一条本轮综合记录

---

## 5. 字段对齐方式

如果你的飞书表字段名与项目默认字段名完全一致，可以直接写。

如果不一致，建议提供一份映射关系，规则是：
- 左边：项目输出字段名
- 右边：你飞书表中的真实字段名

尤其要确认：
- 客户信息的主键字段是否就是 `客户ID`
- 商机快照中时间字段是否真的是“日期时间”类型
- `Lead Score` 是否为数字字段
- `高净值优先` 是否为复选框 / 布尔字段
